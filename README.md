# 自治体向け健康管理システム 保守・品質改善業務

## なぜこのポートフォリオを作ったか

自治体向け健康管理システムの保守・懸案対応を担当する中で、
「動いてはいるが、品質が安定しない」
「受入テストをしているのに、現場で不具合が多発する」
という課題に直面しました。

本ポートフォリオでは、
そうした状況下でどのように仕様を理解し、
品質改善に取り組んだかを整理しています。

## 背景・課題

プロジェクトには以下の構造的な課題がありました。

- **仕様のブラックボックス化**: 設計書が複数バージョン存在し、どれが最新の正解仕様か判断できない状態
- **ドキュメントの品質不統一**: 自社メンバーと外注パートナー間で設計書の粒度や記述ルールが統一されていない
- **テストプロセスの形骸化**: 受入テストの成果物に対するレビュー習慣がなく、テストの妥当性が検証されていない
- **非効率な調査フロー**: 「動かない」という曖昧な報告が多く、再現手順が不明確なまま調査がスタートしていた

## 自分の立ち位置・担当範囲

- **役割**: システム保守・運用、品質改善担当
- **担当範囲**:
  - 問い合わせ調査・回答
  - 不具合発生時の一次切り分け・原因調査
  - 仕様整理およびドキュメント整備
  - 受入テスト設計・実施

## 実施した対応内容

仕様の曖昧さを排除し、「確実なテスト」を行える土台を作ることに注力しました。

1. **三点突合による仕様の再定義**
   - 既存の「設計書」、実際の「ソースコード」、本番環境の「ログ」を突き合わせ、現状の挙動を正解として仕様を逆算・文書化しました。

2. **データ連携機能（CSV取込）のテスト観点整理**
   - 不具合の温床となっていたCSV取込機能について、バリデーション仕様や異常系ケース（必須漏れ、型違反、重複など）を網羅的に洗い出しました。

3. **チェックリスト作成とエビデンス取得の義務化**
   - テスト実施者個人の勘に頼らないようチェックリストを作成。テスト結果には必ずスクリーンショットやログなどのエビデンスを紐づける運用ルールを整備しました。

4. **再現条件の明確化と修正依頼の標準化**
   - 問い合わせ対応時、単に開発へ丸投げせず、必ず手元で再現条件を特定。発生条件・手順・期待値を整理した上で修正を依頼するフローを確立しました。

## 工夫した点

### 1. 「事実」に基づいた仕様把握
古いドキュメントや人の記憶を鵜呑みにせず、常にコードとログという「事実」をベースに判断しました。これにより、「言った言わない」の議論を減らし、技術的な裏付けのある仕様整理を実現しました。

### 2. 再現性の確保による手戻り削減
「なんとなく動かない」レベルの不具合報告を開発者に渡すと、調査コストが膨大になります。
「このデータのこの項目が空の時に、この操作をすると発生する」レベルまで条件を絞り込むことで、開発者が修正作業に集中できる環境を作りました。

### 3. テストの「質」の可視化
テスト項目書を作成する際、「何を確認すればOKと言えるか（合格基準）」を言語化しました。これにより、レビュー時にテストの抜け漏れを指摘しやすくなり、リリース後の手戻りを抑制しました。

## 学び

- **仕様理解とテスト設計が品質の要**：
  保守・品質改善において最も重要なのは、コードを書く前段階の「仕様の正確な理解」と「網羅的なテスト設計」であると痛感しました。

- **曖昧さがテストを形骸化させる**：
  期待値（正解）が曖昧なままでは、テストは単なる「操作確認」になってしまいます。設計を明確にしない限り、品質は保証できないことを学びました。

- **再現条件の言語化が調査効率を左右する**：
  複雑な不具合も、条件を一つずつ切り分けて言語化することで、解決までの時間を大幅に短縮できることを実体験として学びました。
