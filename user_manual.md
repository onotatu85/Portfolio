# Personal User Manual

## 1. 基本的な働き方のスタンス

- 指示された作業について、まず前提条件や目的を整理することを重視しています
- 仕様や背景が不明確なまま進めることで手戻りが発生することを避けたいと考えています
- 保守・運用業務では、安定稼働と再発防止を最優先に行動します

## 2. 「分からない状態」の捉え方

- 経験や知識が不足している状態では、判断できないのは自然なことだと考えています
- 判断できない場合は、その理由（前提知識不足／仕様不明確など）を明確にした上で共有します
- 「何が分からないのか」を言語化することも、業務を前に進める一歩だと捉えています

## 3. 情報整理・調査の進め方

調査を行う際は、以下を整理することを意識しています

- 前提条件
- 再現手順
- 期待される結果

設計書の記載だけに依存せず、コードやログなど事実ベースの情報を確認します
自分なりに整理した内容は、後から他の人が見ても分かる形で残すよう心がけています

## 4. 相談・確認のタイミング

以下のような場合は、早めに相談・確認を行います

- 判断に必要な前提情報が不足している場合
- 自分の判断で進めると影響範囲が大きい場合
- 一定時間（目安として30分程度）考えても方向性が見えない場合

相談時は、「現状の理解」「不明点」「自分なりの考え」をセットで共有するようにしています

## 5. コミュニケーションの取り方

- 基本的にはテキストベース（Issue／チャット）でのやり取りを好みます
- 状況共有は、箇条書きや見出しを使って簡潔にまとめます
- 作業の進捗や状態が分からなくならないよう、定期的な共有を意識しています

## 6. 進捗・状態共有について

- 「完了／未完了」だけでなく、現在どの段階にいるか（調査中・整理中・確認待ちなど）を共有します
- 依頼元の方が「今、何をしているか分からない」状態にならないよう意識しています
- 判断待ち・確認待ちの状態も、進捗の一部として共有します

## 7. 苦手なこと・お願いしたいこと

- 仕様や背景が口頭のみで共有される場合、認識のズレが生じやすいため、可能であれば簡単なメモやテキストでの共有をお願いしています
- 優先度や判断基準が不明確な場合は、補足していただけると助かります
